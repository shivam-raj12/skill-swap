# SkillSwap - Decentralized Learning Platform

## Overview

SkillSwap is a peer-to-peer skill exchange platform built with Next.js 15, enabling users to trade knowledge without financial barriers. The application connects people globally who want to teach skills they have in exchange for learning skills they need. It features real-time messaging, intelligent matching algorithms, and a comprehensive skill taxonomy covering technology, creative arts, business, languages, and more.

## User Preferences

Preferred communication style: Simple, everyday language.

## System Architecture

### Frontend Architecture

**Framework**: Next.js 15 (App Router)
- **Rationale**: Leverages React Server Components for improved performance and SEO
- **Styling**: Tailwind CSS v4 for utility-first responsive design
- **Font Optimization**: Uses Next.js font optimization with Inter font family
- **TypeScript**: Strict typing enabled for type safety across the codebase

**Component Structure**:
- **Page Components**: Located in `/app` directory following App Router conventions
- **Reusable Components**: Organized in `/components` directory with specialized Dashboard subfolder
- **Layout Pattern**: Root layout wraps all pages with global styles and smooth scrolling

**State Management**:
- **Custom Hooks**: Centralized logic in `/hooks` directory for authentication, chat services, and user data
- **Local State**: React useState/useEffect for component-level state
- **Caching Strategy**: localStorage used for dashboard view persistence and profile caching

### Backend Architecture

**Backend-as-a-Service**: Appwrite
- **Rationale**: Provides authentication, database, storage, and real-time capabilities without managing infrastructure
- **Alternatives Considered**: Firebase (chosen Appwrite for better self-hosting options and open-source nature)

**Database Design**:
- **Collections**:
  - `profiles`: User profile data including bio, skills, and profile pictures
  - `conversations`: Chat conversation metadata with participant tracking
  - `messages`: Individual message documents linked to conversations
  - `userskills`: Skill-related user data (referenced but not fully implemented)

**Storage**:
- Profile pictures stored in Appwrite Storage bucket
- File uploads handled with permission-based access control

### Authentication & Authorization

**Authentication Provider**: Appwrite Account service
- **Session Management**: Cookie-based sessions managed by Appwrite SDK
- **Email Verification**: Built-in verification flow with custom landing page at `/verify-email`
- **Protected Routes**: Custom `ProtectedRoute` component wraps authenticated pages
- **Auth Hook**: `useAuth` hook provides global user state and loading status

**Authorization Pattern**:
- Public pages: Landing page, login, register, email verification
- Protected pages: All `/dashboard/*` routes require authentication
- Automatic redirection logic prevents logged-in users from accessing auth pages

### Real-Time Messaging System

**Architecture**:
- **Conversation Model**: Deterministic conversation IDs generated by sorting participant IDs (`userA_userB`)
- **Message Storage**: Messages linked to conversations with sender tracking
- **Chat Service Hook**: `useChatService` manages message fetching and sending
- **Conversations Hook**: `useConversations` aggregates conversation summaries with participant details

**Pros**:
- Simple deterministic ID generation prevents duplicate conversations
- Direct database queries without complex real-time subscriptions initially

**Cons**:
- Currently lacks real-time updates (would benefit from Appwrite Realtime in future)
- Manual polling or refresh required to see new messages

### Skill Matching System

**Hierarchical Skill Taxonomy**:
- Organized in `/constants/skills.ts` with nested categories
- Categories: Technology, Creative Arts, Business, Languages, Health & Wellness, Trades & Crafts
- Sub-categories and granular skill selections available

**Matching Algorithm**:
- Mutual benefit matching: Finds users where skillsToTeach of one matches skillsToLearn of another
- Bidirectional matching ensures both parties benefit from exchange
- Profile filtering excludes current user and caches results for performance

### Image Upload & Management

**Storage Solution**: Appwrite Storage
- **Bucket**: Dedicated profile picture bucket
- **Upload Flow**: 
  1. File validation on client side
  2. Upload to Appwrite Storage with user-specific permissions
  3. Store file URL in profile document
  4. Previous image cleanup on updates

**Permission Model**: Files created with user-level read permissions and owner update/delete permissions

## External Dependencies

### Third-Party Services

**Appwrite** (v20.1.0)
- **Purpose**: Primary backend infrastructure
- **Services Used**:
  - Account: User authentication and session management
  - Databases: Document storage for profiles, messages, and conversations
  - Storage: Profile picture and file uploads
- **Configuration**: Requires environment variables for endpoint and project ID

**Resend** (Email Service)
- **Purpose**: Email delivery for verification and notifications
- **Integration**: API key configured in constants (not fully implemented in shown code)

**Next.js** (v15.5.4)
